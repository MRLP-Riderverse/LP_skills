# Acadie.sol Website Development

**Domain**: Front-end development for the Acadie.sol community site (HTML/CSS/JS) following OS/product interaction design principles.

## Trigger Conditions
- Working in `~/ExoCortex/websites/projects/acadie_sol/` or any subdirectory thereof.
- Tasks involve editing `index.html`, `directory.html`, `events.html`, `entry.html`, associated CSS (within `<style>` blocks), or JavaScript.
- User wants low-friction, raw-preserving drafts and clear, modular code with explanatory comments.

## Workflow
1. **Edit with intention**
   - Make small, focused changes.
   - Add a comment block above each logical section explaining its purpose and how it relates to neighboring sections (e.g., “// Dock layout – glassy nav at bottom, mirrors OS product interaction”).
   - Keep markup semantic; avoid unnecessary wrappers.
   - Prefer raw-preserving drafts: store incomplete ideas in comments or a `// TODO` block rather than over‑structuring.

2. **Local verification**
   - Start a local server: `python3 -m http.server 8777` (from the project root).
   - Open `http://localhost:8777/index.html` on mobile or use puppeteer to screenshots.
   - Run a quick JS sanity check:
     ```bash
     for html in *.html; do
       node -e \"const fs=require('fs'); const c=fs.readFileSync('$html','utf8'); 
       const scripts=c.match(/<script>[\\s\\S]*?<\\/script>/g); 
       if(scripts) scripts.forEach((s,i)=>{try{new Function(s.replace(/<\\/?script>/g,''))}catch(e){console.error('$html script'+i+':',e.message)};})\"; 
     done
     ```
   - Ensure no console errors in the browser.

3. **Check for common issues**
   - Run `git diff --check` to detect trailing whitespace.
   - Verify that all `var(--token)` usages are defined in the `:root` and `[data-theme="dark"]` blocks.
   - Confirm that any hard‑coded colors have been replaced with design tokens where possible.
   - If `scripts/export_to_site.py --all` rewrites `.ics` calendar files without event source changes, inspect the diff; timestamp-only ICS churn can trip `git diff --check`, and line-ending churn can masquerade as trailing whitespace. Revert `assets/calendar/` before committing if those files are not meaningful to the change.
   - Current root cause of the `.ics` churn: `scripts/export_to_site.py` writes `DTSTAMP` from `datetime.now(...)`, so every export rewrites calendar files even when events did not change. Treat `DTSTAMP`-only diffs as noise, verify with `git diff -- assets/calendar/*.ics`, and revert those files unless event fields like `SUMMARY`, `DTSTART`, `DTEND`, `DESCRIPTION`, or `LOCATION` also changed. See `references/export-ics-dtstamp-churn.md` and `references/export-and-calendar-verification.md`.

   - For color/contrast tasks, explicitly audit links, descriptions, notes, footer text, placeholders, quick cards, drawers, and secondary pages (`recents.html` too). The user's standard is direct readability: dark text on offwhite in light mode, light text on charcoal in dark mode.
   - When light/dark palette tokens drift across pages, centralize the palette into a shared asset such as `assets/site-colors.css`: one `:root` block, one `[data-theme="dark"]` block, compatibility aliases for older page tokens, and one stylesheet link before each page-local `<style>`. Remove page-local token declaration blocks instead of continuing per-page tweaks. See `references/global-theme-token-shell.md`.
   - Treat button chrome as a *global system* once the shell starts sharing it. Use shared tokens (`--button-bg`, `--button-bg-hover`, `--button-fg`, `--button-border`) so dock items, drawer buttons, and CTA buttons stay visually aligned across modes. See `references/global-button-system-and-dark-mode-blue.md`.
   - If the user asks for “more glassy” or “solid, not gradient” dark mode, treat that as a site-wide surface pass: remove decorative radial/linear gradients from page backgrounds first, then restore contrast with solid `--page-bg` / `--paper` / `--card` surfaces and slightly higher-transparency glass tokens (`--glass`, `--button-bg`, `--button-bg-hover`). See `references/solid-surfaces-and-glass-tuning.md`.
   - When a link or action reads too dark in dark mode, check the actual token on the element (`var(--link)` or a page-local alias) instead of assuming the shared shell covers it. Page-local CTA classes often need to be moved off `--blue-deep` and onto `--link` / `--link-hover` for accessibility.
   - Search for stale rules like `a { color: inherit; }`, `color: #fff`, `color: rgba(...)`, old `--text` token usage, and old duplicate/dangling `:root` / `[data-theme="dark"]` blocks before calling the palette unified.
   - When text still “looks hard to read,” verify with browser computed styles for the exact selectors the user named; visual screenshots alone are not enough.
   - For responsive/layout complaints, test phone, iPad portrait, iPad landscape, and desktop viewports with browser-computed widths. Watch for accidental global mobile-shell overrides like `width: min(430px, 100vw)` outside a mobile media query; tablet/desktop should use available canvas and readable card widths.
   - For mobile text inputs that zoom on focus, keep the control at 16px minimum and add `html { -webkit-text-size-adjust: 100%; text-size-adjust: 100%; }` before blaming the layout. This is a browser threshold issue, not a design-token issue.
   - If a top header / nav layer is obsolete for the page, delete the DOM/CSS/JS entirely instead of only hiding it. Hidden stale chrome still keeps dead code around and can leak into tablet/touch layouts later.
   - If the controls still belong on desktop but not on touch, prefer a touch-pointer media query (`@media (hover: none) and (pointer: coarse)`) over a width-only rule.
   - If a full-page card/action opens inside a nested shell or expandable summary, route it as a top-level navigation target (`target="_top"` plus a normal href) so the full page does not inherit weird intermediate card state.
   - For homepage IA tweaks, do not leave the primary CTA as just another neutral paper card if the user asks for stronger differentiation. Prefer a clickable banner/launch strip treatment (e.g. marquee band, stronger gradient, dedicated border/light treatment) so the main route reads as a gateway rather than another container.
   - If you use a marquee/banner CTA, make it a single unified clickable surface. Do not stack a marquee strip on top of a second button/card that repeats the same action — that reads as two overlaid UI layers instead of one launch strip. Put the marquee and the CTA body inside the same anchor/container.
   - If that still feels visually split on mobile, demote the marquee from a visible top band into a low-contrast background texture inside the same CTA surface: absolute-position it behind the body copy, remove divider lines, and keep one continuous click target.
   - When the user asks for a hero to feel more like a banner/panorama, reduce vertical height materially (about 20–30%) before redesigning the whole section; this usually lifts the lower UI into view with minimal markup churn.
   - When the homepage feels overwhelming and the bottom dock already owns Directory/Search and Events navigation, remove duplicate homepage directory/event widgets instead of making them prettier. Prefer one welcome card plus one calm future-space CTA/placeholder (calendar/gallery/community feature), and remove home-level JSON fetches if the widgets no longer need live data.
  - For homepage simplification, do not stack the same positioning idea twice in the top copy (for example repeating "local-first pocket..." in both subtitle and body). The user's tolerance for landing-page copy is low: if the page starts reading like “blahblah,” cut it aggressively until only the identity line, one short slogan, and the necessary CTA layer remain.
  - Treat the landing banner as a centered identity block when the user asks for a cleaner welcome surface: central justification, restrained title sizing, and enough internal padding (about 20px) to feel like a banner rather than a left-aligned article card.
  - If the hero image/container is removed, do not just delete the meaning with it. Re-home the crucial framing into a short built-locally block separated by a divider line inside the top banner if the user wants the footer removed, or into footer bullets if they explicitly want the framing lower.
  - When the home CTAs (for example Events and Search places) are primary routes rather than explanatory copy, give them their own container/surface directly below the welcome block. This keeps the landing text readable and the action layer visually separate.
  - **Current Home attention hierarchy:** mobile uses three visible discovery sections in order: quiet identity → one owner-controlled current/LIVE signal → one recently added album. Entry and event counts in the welcome section are direct links, so they do not need duplicate global routes. Center the welcome identity block vertically like the other cards while keeping the swipe cue anchored near the panel bottom. Discord remains a static action on `community.html`; do not add a floating Discord prompt to Home unless the user revisits the concept. Desktop presents the same three sections without hiding or auto-rotating them: identity left, current/recent stacked right.
  - **Current global shell IA:** mobile has no route dock. Use one 58px bottom-right three-line Menu launcher; Menu contains only Home, Search, Photos, and Extras. The full-page `extras.html` hub owns Directory, Events, Updates, Community, Support, About, and Remembrance. Desktop keeps Home/Search/Photos as primary routes plus the same focused Menu. The narrow global mobile banner participates in normal document flow (`position: relative`) rather than floating over page content. Preserve this separation so future focused features can enter Menu without restoring every destination to global chrome.
  - If the user says Home feels cluttered on mobile while desktop feels good, make a restrained mobile-only pass rather than redesigning both. Remove duplicate top chrome, flatten the welcome surface, keep LIVE as the only dominant card, shorten the photo card, hide redundant descriptions/actions, and turn the archive exit into a quiet row.
  - **Mobile discovery deck vs. feed:** when the user wants Home to feel like “let’s find something you like” rather than a top-to-bottom feed, isolate the experiment below `900px` and let native horizontal swiping own discovery. The current contract is three top-level semantic sections (welcome, current notice, recently added), not a nested carousel: each mobile panel is `100vw`, borderless, backgroundless, shadowless, and represented by a localized bottom position dot. On desktop, keep welcome left and stack current/recent as separate right-column sections. Position dots must update from real scroll position and remain tappable with enlarged invisible hit targets. Do not force `touch-action:pan-x`; short/landscape panels need `overflow-y:auto` so their final CTA remains reachable while document scrolling is disabled. Treat this as a reversible, test-by-feel design experiment. See `references/mobile-home-discovery-deck.md`.
  - **Page-coherent slogans:** do not remove a phrase such as `Vive l’Acadie!` merely because the banner feels out of sync. First make the banner inherit the page surface language (`var(--paper)` + `var(--ink-strong)` with shared line/blur), then let the user judge the phrase in context. Avoid branded gold over a semi-transparent medium-blue surface unless measured contrast passes. See `references/mobile-home-discovery-deck.md`.
  - Duplicate navigation is clutter: when a homepage count or label already links directly to Directory or Events, do not add those routes back to the mobile shell. Keep secondary destinations in Extras unless the user explicitly promotes one into the focused Menu.
  - If Home feels “overinflated” or text-heavy, cut copy before changing architecture. Prefer one short identity line and one necessary CTA per secondary surface.
  - Use `assets/live.json` as the small owner-controlled attention signal. Use the dedicated `home-feed.html` archive as the quiet historical exit; do not embed the archive application into Home.
  - Keep the mobile decluttering breakpoint aligned with the shared shell (`<900px`) and verify visibility at 320, 390–430, 820, 900, and 1280px. See `references/homepage-attention-hierarchy-and-mobile-declutter.md`.
  - Treat homepage live modules as localization-sensitive renderers, not static copy. If card titles/summaries/badges are normalized from JSON, re-run the data normalization on `acadie:languagechange` instead of only repainting stored localized strings; otherwise the shell flips languages while the feed/carousel content stays in the old language until reload.
  - Do not hard-fail Home or `home-feed.html` just because `assets/site-meta.json` is missing or stale. Require the real content payloads (`directory-data.json`, `events-data.json`), but treat `site-meta.json` as optional enhancement for latest-update stamps.
  - If the user pushes the homepage routes toward a rawer tone, strip the CTA chrome aggressively: no pill/button surface, no extra wrapper card unless it is structurally necessary, and prefer simple stacked text links with underlines over faux-button styling.
  - If those bottom-of-page home links are supposed to sit near the dock, verify the layout primitive itself before tweaking margins. In the current homepage shell, `margin-top: auto` only works reliably once `.phone-shell` is a flex column; leaving it as a grid can make the links look "moved in code" while still floating mid-page.
  - Decorative divider treatment requested by the user can be intentionally literal. Reusing a rune divider such as `✧❅✦❅✧` above both the top metadata block and the dock-near links is preferable to reintroducing bordered containers when the user wants a raw textual feel.
  - For Acadie.sol homepage CTA labels, prefer short route names over descriptive phrases when the user is tuning tone. `Everyone` can point to the directory `View All` route (`directory.html`) instead of acting like a search affordance.
  - Dynamic event counts on the homepage CTA are optional, not sacred. If the page already exposes the event count in a stronger global header/meta position, remove the duplicate number from the local `Events` link instead of echoing `$N` twice.
  - When the user wants the homepage routes to feel extra raw, stack the action links as plain underlined text directly in the welcome/meta flow (`Events` over `View All`/`View Everyone`) with no extra wrapper surface, pill shape, or button chrome unless a structural container is genuinely required.
  - When the user asks for the pixel poke font on the homepage banner, wire the local `assets/fonts/PokemonGb-RAeo.ttf` asset via `@font-face` and apply it selectively to the small header/prefix and the `Acadie.sol` wordmark, not to all body copy.
  - **Pixel font is too wide for mobile nav text**. The Pokemon GB font renders significantly wider than system monospace at the same `font-size`. For fixed-width elements like the global `.site-header` banner, use `font-family: monospace` instead of the pixel font — the user confirmed monospace fits mobile viewports while the pixel font does not. Keep the pixel font for decorative uses (welcome card title, section pills, labels) but not for constrained horizontal chrome.
  - **"Bulge pill" border pattern**: when a label pill should straddle a container's border line, set `position: absolute` on the pill within the positioned container, then use `bottom: 0; left: 50%; transform: translate(-50%, 50%)` (bottom border) or `top: 0; left: 50%; transform: translate(-50%, -50%)` (top border). The pill center aligns exactly with the border. Horizontal centering via `left:50% + translateX(-50%)` is preferred over fixed `left` offsets so the pill stays centered regardless of container width.
  - **Pill below card → gallery collision**: when the pill sits on the bottom border of a surface card, the next sibling in the grid (e.g., media gallery) can collide with the pill. Add `margin-top` (6px minimum) to the next section to create clearance, since `.phone-shell`'s grid `gap` only applies between container edges, not between absolute-positioned children that overflow their container.
  - **Section dividers should include a visual boundary line**: the `✧❅✦❅✧` ornament above the updates feed should be followed by a thin `1px` divider line (using `var(--line)` at reduced opacity) to give users a clear sense of "this ends, the updates start."
  - **Global header body padding**: when the header is 30px tall and the pill/first element needs at least 12px clearance below the header, set `body { padding-top: calc(42px + env(safe-area-inset-top)); }` (42 = 30px header + 12px clearance). This replaces the old `calc(30px + ...)`.
  - **Do not split the header's `height` and `padding` to add clearance.** The combination `height: auto; min-height: calc(30px + env(safe-area-inset-top)); padding: env(...) 0 12px; box-sizing: border-box` was attempted and **caused the home page to load offset/cropped in light/dark and especially in FR mode** (`git revert 2386995` reverted this fix). Keep the safe-area-aware header height as `calc(30px + env(...))` and rely on the matching body padding-top for clearance. When the welcome card slips under the header, suspect paint-timing first, not the CSS primitives.
  - **Update timestamps must step through units past 48h.** The updates feed previously rendered raw hour counts (`71h ago`); replace the simple `mins < 60 ? min : h` formatter with a breakdown function that picks among `min / h / d / w` at thresholds 1h / 48h / 14d. Strings: EN `5 min ago`, `3 h ago`, `3 d ago`, `4 w ago`; FR `Il y a 5 min`, `Il y a 3 h`, `Il y a 3 j` (jours), `Il y a 4 sem` (semaines). The breakdown function is a plain function declaration outside `HOME_COPY` (hoisted) so both EN and FR `updatedAgo` can call it. Verify boundaries (`2880 min → 2 d`, `20160 min → 2 w`) with a `node -e` inline test before pushing.
   - When pruning homepage sections, replace direct `document.getElementById(...).textContent` calls with small guarded helpers like `setText` / `setHtml` before deleting old IDs, so iterative IA cleanup does not create null-element runtime errors.
   - For language-toggle fixes, audit both static DOM copy and dynamic renderer copy. Hero/page headers, drawer subtitles, search placeholders, result counts, empty states, and card detail labels often stay hardcoded unless they get IDs and are wired into the page copy object.
   - When localizing card detail labels, keep a stable machine type separate from the displayed label (`['phone', copy().phone, value]`), otherwise linkification breaks when `Phone` / `Address` become French.
   - Keep the floating Menu as a centered bottom panel opened by the bottom-right three-line launcher rather than a side drawer. Separate `.drawer-backdrop` and `.drawer` checked-state CSS; never apply the drawer `translate(-50%, …)` transform to the full-screen backdrop, or a right-side overlay artifact can appear. Prefer dim+blur backdrop and compact top-corner controls (`☾`/`☼`, `EN / FR`). See `references/site-i18n-and-floating-menu.md`.
     - Closed overlays must be removed from hit-testing, not just faded out. If a hidden fixed drawer or modal still steals taps/clicks, add `visibility: hidden` alongside `opacity: 0` and `pointer-events: none`, then restore visibility only on open. If a page-local link starts routing into the drawer or another overlay, inspect `document.elementFromPoint(...)` before assuming the href changed. See `references/overlay-hit-testing-and-toggle-fixes.md`.
     - The dock/menu is global chrome, not a per-page component. If drift appears, centralize the shell in shared assets (for example `assets/site-shell.css` + `assets/site-shell.js`), remove per-page shell markup/CSS/listeners, and let pages respond to shared shell events rather than owning the controls.
     - Treat the shell breakpoint as a page-layout contract. The fixed 58px Menu launcher remains visible through `899px`; retain enough bottom padding for the launcher and safe-area inset on editorial, gallery, and album shells, then verify it does not cover final-page actions.
     - Menu launchers must be keyboard-operable even when a hidden checkbox owns the CSS state. Give label-based launchers `role="button"`, `tabindex="0"`, `aria-controls`, and synchronized `aria-expanded`; handle Enter/Space, move focus into the drawer on open, trap Tab/Shift+Tab inside it, close on Escape, and return focus to the launcher on every close path. While open, make all non-drawer body siblings `inert` (restoring prior inert state on close) and keep the backdrop/drawer z-index above both mobile and desktop shell navigation.

   - For the `search.html` first-render state, keep the surface clean and non-influential: no preloaded cards, no empty-state, no instruction paragraph, no visible suggestion/chip row. Use a minimal placeholder (`?`), hide filters behind a compact control, and place `View All` / `Filters` above the input. Anchor the input low near the dock/finger zone via `position: fixed`. Avoid cage styling; prefer a transparent underlined input line (bottom‑border only). Keep the page background on the global token (`background: var(--page-bg)`). Font sizing: search input ~50% smaller than hero defaults (`clamp(1rem, 3.4vw, 1.15rem)`) but never below 16px; button/pill text ~10% reduced (`.79rem`). Implement a pill‑shaped search chrome: fully rounded (`border-radius: 18px`) with a thin black border (`#000 at 32% opacity`) and `var(--page-bg)` background, plus a subtle shadow so results scroll behind it with a flash‑card/parallax feel. Results should render in normal top-down reading order, not bottom-up thumb-stacking; keep the active search state top-anchored so cards/empty states start from a stable line below the fixed chrome. Separate results with thin divider lines (`border-bottom`) instead of container cards. Dismiss the keyboard on Enter (`input.blur()`) and on significant downward drag through results. Prefer the simplest stable mobile behavior for the search bar itself: glue `.search-controls` to one bottom lane above the dock, let the browser/viewport move it with the keyboard naturally, and hide/duck only the dock while the input is focused. Treat scroll-hide, custom keyboard-lane math, and viewport-driven bar repositioning as optional extras to remove first when the bar starts "leaving with" the results or jittering during zero/one-result transitions. See `references/search-page-minimal-entry.md`, `references/search-mobile-flow-state.md`, and `references/search-bar-bottom-glue.md`.
   - For Acadie.sol search-page entry state, do **not** add instructional/marketing copy, visible tag rows, preloaded results, or a page-specific branding treatment. The initial surface should be quiet: matching site background, one search input, a compact filter pill that expands on demand, and an optional directory escape hatch. Users know how search bars work; copy should appear only when it resolves a real state such as no results or failures. See `references/dedicated-search-surface.md`.
   - For subtle utility links that belong in the menu but not the dock, prefer an icon-only button with no adjacent visible text when space/sensitivity matter. Put the control beside the EN/FR toggle if it needs to stay discoverable but quiet. See `references/obituaries-section.md`.
   - For remembrance/obituaries pages, keep the public copy comforting and minimal: bilingual headline, one short supportive sentence, and name-only source links (no URL clutter). Use a gentle public title while keeping any internal/admin label simple for recall. See `references/obituaries-section.md`.
   - For homepage event previews, support up to 6 items and include a compact status pill.If an event has started but not yet expired, surface it as live/ending soon instead of hiding it.
   - For homepage countdown copy, prefer terse pills over sentence fragments when space is tight: English `IN 6D` / `IN 8H`, French `EN 6 J` / `EN 8 H`, while still switching to hours for anything under 24h.
   - When standardizing homepage calendar chips, watch for broad text selectors such as `.event-row span` overriding the chip month/day ink. Scope muted copy rules to the text column (for example `.event-row > div span`) and explicitly enforce chip ink on the badge and its descendants so the red/yellow treatment stays readable.
   - For event-list browsing UX, prefer directory-style expandable quick cards with `<details>/<summary>` when the user wants to see more options by default without forcing a full-page jump. Include deep-link hash opening so `events.html#event-…` expands the matching card on load.
   - For side-by-side expandable cards, set the grid container to `align-items: start` and avoid `min-height: 100%` on cards. Otherwise opening one `<details>` card stretches the closed neighbour to row height, creating an “empty card” illusion.
   - **Always-dark component standard**: some UI elements should ignore the light/dark toggle and always render in dark mode. When a component looks great in dark mode but off in light mode, and the user says "declare this a standard," use hard-coded dark colors (`#f3c85b`, `rgba(10, 43, 87, 0.68)`) instead of theme tokens (`var(--acadian-yellow)`, `var(--glass)`). Only the border color stays on a theme token (`var(--red)`) for readability. This was applied to event date badges and may extend to other brand-signal elements.
   - **Archive management via status flags**: to archive events/entries, set `status: "archived"` and `computed_status: "archived"` in the generated JSON data. Update top-of-file count objects accordingly. Archived items leave the "Upcoming" filter automatically and appear under "Archive." The source data stays in the JSON for admin/GitHub access. At scale, need a dedicated archive.json endpoint.
   - **Placeholder detection pattern**: `publicSummary()` and `publicTitle()` in events.html mask placeholder events via regex (`/placeholder|non-public/i` in summary, `/Placeholder/i` in title → replaced with "Preview"). To mark a real event as placeholder without deleting it, seed the word "placeholder" into the summary text across all locales.

   - If an event venue lacks a polished directory entry, add a reusable `locations/<location-id>/` record so the public calendar can ship without blocking on full entry polish. If users need to click through to the venue, promote/create `entries/<venue-slug>/` too and set both `event.host_entry_ids` and `location.entry_ids` to that slug.
   - Event host, venue, and “View host” links should target the full entry route `entry.html#<slug>`, not `directory.html#entry-<slug>` quick cards, whenever the target is a published entry. Keep quick-card hashes only for directory-internal expansion flows or drafts.
   - When promoting an inbox draft into a published entry, remove the old `inbox/<slug>.md` draft before export; otherwise the exporter can emit duplicate slugs and break hash routing.
   - Export with `python3 scripts/export_to_site.py --all --site ~/ExoCortex/websites/projects/acadie_sol` from the directory repo.
   - After export, expect the site repo to change even when no page code changed: `assets/directory-data.json` always refreshes, and `assets/site-meta.json` usually refreshes with a new `generated_at`, source commit, and snapshot count.
   - Verify the generated payload chain in the website repo, not just the top-level directory payload: `assets/directory-data.json`, `assets/events-data.json`, `assets/locations-data.json`, `assets/search-index.json`, and `assets/site-meta.json`.
   - For event-driven exports, confirm the new event appears in both `events-data.json` and `search-index.json` before declaring the site up to date; use the event slug/id as the primary lookup key.
   - If a public business/event is reported missing, search prior sessions first for the earlier provenance and public-source notes before reconstructing it. Then create the source record, export again, and verify the generated payloads rather than hand-editing the site mirror.
   - To remove an event from the active public calendar without losing the record, set `status` to `archived` in `acadie_sol_directory/events/<event-id>/meta.json` and re-export. Then verify the generated website payloads show the archive state consistently (`assets/events-data.json` counts/status, `assets/search-index.json` badges/subtitle, and `assets/calendar/<event-id>.ics` only if event timing/details changed).
   - Verify generated `assets/events-data.json`, `assets/locations-data.json`, `assets/search-index.json`, and `assets/calendar/<event-id>.ics` in the website repo.
   - Render practical details like `bring` and `wayfinding` from event metadata via reusable event-card fields; do not hardcode event notes into `events.html`.

5. **Commit**
   - If the user supplies an exact commit message, use it verbatim; explicit wording overrides the conventional-commit default.
   - Otherwise write a clear conventional commit message:
     `type(scope): short description`
     e.g., `feat(dock): reorder to Home→Events→Search→Menu, add glassy backdrop`.
   - Selectively stage intended paths. Never use a broad add when unrelated local photos/media are untracked.
   - If the same local candidate media must remain excluded across sessions, add explicit path entries to `.gitignore`; selective staging protects the current commit, while ignore rules protect future broad staging.
   - If the change affects user-visible copy or design, seek approval before committing or pushing (the user prefers approval before commit/push).

6. **Push and prove deployment**
   - `git push` to trigger GitHub Pages.
   - Verify local `HEAD` equals the remote branch SHA.
   - Verify the latest Pages build references that SHA and reaches `built`.
   - Fetch `https://acadie.sol.site/?v=<short-sha>` and check a unique marker from the change; HTTP 200 alone can still be the previous deployment.

7. **Post-push review**
   - Open the live site on mobile to confirm appearance and interactions.
   - If anything looks off, revert or follow‑up with a fix.

## Pitfalls & How to Avoid Them
- **Homepage signal hierarchy**: Keep the homepage artifact-first and maintenance-light: integrated public counts, a calm inactive LIVE fallback, manifest-derived **Recently added** content, quiet opt-in community access, and operational/project-log detail beside directory/search data rather than in the primary home composition. Follow `references/homepage-signal-carousel.md` for the complete interaction and QA contract.
- **Do not nest a carousel inside the mobile discovery deck:** the current Home model uses three top-level sections with no autoplay. A previous two-slide attention carousel created a second interaction layer and made mobile feel like a card placed over a background. Keep current/recent independently visible on desktop and independently swipeable on mobile. If a future page genuinely needs autoplay, preserve the older durable-pause contract in `references/homepage-signal-carousel.md`; do not import that machinery back into Home by default.
- **Over‑engineering drafts**: Resist the urge to create elaborate wrappers for placeholder content. Keep drafts as simple as possible; use comments to capture intent.
- **Search page as directory clone**: The search page is a quiet input surface, not a card-based dashboard. Keep it clean: minimal placeholder (`?`), compact filter control, `View All`/`Filters` above input, fixed low chrome, pill‑shaped search chrome with thin black border, and top-anchored results with divider lines. Do not reintroduce preloaded cards or center-grown result layout. See `references/search-page-minimal-entry.md`.
- **Search keyboard/focus loop**: On mobile, `position: fixed` inputs + soft keyboards create a focus-blur cycle if you blur on every scroll event, listen to `visualViewport.scroll`, or force `input.focus()` after filter selection. Use a `focusGuard` flag (150ms window after focus), debounce keyboard detection (80ms), and never re-focus the input programmatically. Also treat focused editing as its own state: while the search input is focused, keep the page in active/top-anchored search flow even if the query becomes empty or yields zero/one results, and let focused-input / keyboard-open state override scroll-hide so the bar snaps back above the keyboard. For dock stability on iPhone, hide the dock off `.search-input-focused` as the primary truth, with `keyboard-open` as a fallback only; `visualViewport` keyboard detection can wobble during backspacing/editing and should not be the sole state gate. Critically: do **not** blur the field from generic `window.scroll` deltas while editing. iPhone keyboard/layout changes can emit fake scroll movement during backspacing and cause search/dock jitter. While the field is focused, ignore scroll-direction chrome mutations and keep the chrome visible; if you want swipe-down-to-dismiss, gate blur off a real touch-drag threshold (`touchstart`/`touchmove`) rather than scroll events. Also do not synthesize `keyboard-open` on `focus`—let `visualViewport.resize` establish it for real. See `references/search-page-minimal-entry.md` → "Mobile keyboard / focus conflict" and `references/search-mobile-flow-state.md`.
- Search keyboard lane baseline: If the fixed search bar still jumps on iPhone while backspacing or shrinking from many results to one/zero, the usual culprit is keyboard detection tied to the live current `window.innerHeight`. Do not compare `visualViewport.height` against a wobbling live layout height. Keep a stable keyboard-closed layout-height baseline, compute a real `keyboardInset` from `visualViewport.height + visualViewport.offsetTop`, and feed that inset into CSS (for example `--keyboard-inset`) so focused input state gets a dedicated lane above the soft keyboard. Also reserve bottom space in active results so sparse/empty result transitions do not visually fight the fixed bar, and verify swipe-dismiss uses a real downward delta (`currentTouchY - touchStartY > threshold`). If the bar still vanishes in iOS/PWA after simplifying the JS, inspect the CSS primitive itself: a `position: fixed` element that is also centered with `left: 50%` plus `transform: translateX(-50%)` can disappear during viewport/keyboard chaos. Prefer a plain fixed outer shell (`left: 0; right: 0; display: flex; justify-content: center`) with a normal inner card, or otherwise remove transforms from the fixed layer before attempting a larger layout rewrite. See `references/search-mobile-keyboard-lane-and-pwa-path.md`.
- **Filter chips `top` inheritance**: When overriding `.chips` from `position: absolute` to `position: fixed`, the base `top` rule leaks through. Always add `top: auto` in the `[open]` override or `bottom` will be ignored and chips appear below the viewport.
- **Missing comments**: Future maintainers (including you) will lose the reasoning behind a section. Always add a comment that explains *why* the block exists and how it ties to the broader layout.
- When adding new colors, always add them to both `:root` and `[data-theme=\"dark\"]` blocks; never hard‑code a hex value without a token.
- For button chrome unification, define shared `--button-bg`, `--button-fg`, and `--button-border` tokens once in the global palette and reuse them across drawer controls, dock buttons, and page-local CTAs.
- **Malformed or duplicate dark blocks**: A dangling `[data-theme=\"dark\"]` near the end of an inline stylesheet can silently break later contrast assumptions. Search for duplicate/dangling theme blocks before declaring a palette fixed.

- **Skipping local verification**: A change that looks fine in the editor may break layout or JS in the browser. Always run the local server and check for console errors.
- **Pushing without approval**: The user likes to review changes before they go live. If unsure, ask for a quick confirmation.
- **GitHub Pages PWA launch path**: For installed/home-screen behavior on GitHub Pages project sites, do not use root-absolute manifest paths like `"/index.html"` for `start_url`. That can launch the app at the domain root and produce a 404 while Safari browsing still works. Prefer relative manifest values such as `"./index.html"` with `"scope": "./"` so the installed app stays inside the repo subpath. Reinstall the homescreen shortcut after changing manifest metadata because iOS caches icons and launch settings aggressively.
- **Search UX half-state after mobile fixes**: If you abandon the bottom-fixed search shell to stop iOS/PWA viewport conflicts, do not leave search in a semi-static “top until scroll/hide weirdness” state. That reads as accidental browser behavior, not intentional product design. Make a deliberate choice: either build a true top sticky search header with results flowing directly underneath, restore a stable bottom-glued search model, or — when the user's priority is simply a stable working environment — strip the theatrics entirely and use a plain top-native search bar with controls first and results directly below. Avoid compromise states that feel Houdini-ish.
- **Filter overlay should be a real modal**: when the user asks for a filter popup with a backdrop, move it outside any clipped search/control panel and make the launcher toggle open/closed. If the first click only opens and the user must use a separate close icon, the control is too modal for its own launcher; make the launcher itself close it too. Use backdrop click + Escape as additional exits.
- **Entry thumbnail pipeline**: If the user supplies a business photo and wants it used as an entry thumbnail, wire it through the source-of-truth flow instead of patching generated JSON by hand. Store the file under the site repo (for example `assets/entries/<slug>/thumbnail.jpg`), put a small `thumbnail` object (`src`, `alt`) in the directory entry `meta.json`, pass `thumbnail_src` / `thumbnail_alt` through `scripts/export_to_site.py`, and render the image in `entry.html` with the initial-letter poster as fallback. Verify by exporting, checking `assets/directory-data.json`, and running exporter tests.
- **Directory quick-card thumbnails are a separate renderer**: Wiring `thumbnail_src` into the full entry page does not automatically populate `directory.html`. The quick-card summary renderer and the full entry renderer each need to read the exported thumbnail fields explicitly. For current Acadie.sol behavior, keep the square thumbnail/logo only in the collapsed summary rail and remove duplicate expanded-body copies; keep the letter-badge gradient as the fallback when no thumbnail exists.
- **Corner wordmark removal is multi-page chrome work**: If the user says the top-left `acadie.sol` label can go, audit every page-specific shell (`index.html`, `directory.html`, `entry.html`, `events.html`, and any similar secondary pages) rather than deleting it from one template and assuming the rest inherited the change.
- **Shared shell copy must stay in sync**: when `assets/site-shell.js` gains a new drawer item or renamed route, search page-local copy blocks for stale labels (`Full directory`, old drawer subtitles, etc.) and update the shared sync routine at the same time so the drawer render and the hydration path cannot drift.
